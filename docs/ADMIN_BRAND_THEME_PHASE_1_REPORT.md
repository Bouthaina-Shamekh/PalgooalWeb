## ٢. Value Object

**الملف:** `app/Support/AdminBrand/AdminBrandThemeSettings.php`

```php
AdminBrandThemeSettings::fromArray(array $data): self
AdminBrandThemeSettings::defaults(): self

->primary():   string   // --admin-color-primary
->secondary(): string   // --admin-color-secondary
->muted():     string   // --admin-color-muted
->body():      string   // --admin-color-body
->custom1():   string   // --admin-color-custom-1 (empty = not emitted)
…->custom5()

->toArray(): array      // round-trip back to JSON
```

**Sanitizers:**
- `sanitizeHex()` — validates `#rrggbb` / `#rgb`, falls back to default if invalid
- `sanitizeHexOrEmpty()` — same but returns `''` for custom slots (absent = not emitted in CSS)

يتبع نفس نمط `TenantThemeSettings` تماماً.

## ٥. Loading Flow

### ترتيب تحميل CSS في head.blade.php

```
1. app.css        ← @theme aliases: --color-purple-brand: var(--admin-color-primary, #240a37)
2. admin-brand.css ← :root { --admin-color-primary: #240a37; }
3. purple_topbar_css (dynamic route)
4. palgoals_marketing_footer_css (dynamic route)
5. tenant-theme.css (per subscription, if exists)
```

**الكود في head.blade.php:**
```blade
<link rel="stylesheet" href="{{ mix('assets/tamplate/css/app.css') }}">
@php $_adminBrandCssUrl = \App\Support\AdminBrand\AdminBrandCssGenerator::publicUrl(); @endphp
@if ($_adminBrandCssUrl)
    <link rel="stylesheet" href="{{ $_adminBrandCssUrl }}">
@endif
```

### لماذا هذا الترتيب مضمون

CSS cascade: كلا الملفَين يُعرِّفان على `:root` بنفس specificity. الملف الثاني (`admin-brand.css`) يُحدِّد `--admin-color-primary` → يُلغي الـ `undefined` → `var()` في `app.css` يستخدم القيمة الجديدة بدلاً من الـ fallback.

## ٦. Initial Generation

في `AppServiceProvider::boot()`:

```php
$this->app->booted(function () {
    AdminBrandCssGenerator::generateIfMissing();
});
```

**السلوك:**
- أول تشغيل للتطبيق → الملف غير موجود → `generateFromDb()` → إذا لم يُحفظ شيء بعد → `DEFAULTS` تُستخدم → الملف يُنشأ بالألوان الافتراضية
- كل تشغيل لاحق → `Storage::exists()` يُعيد `true` → لا عمل إضافي (near-zero overhead)
- أي خطأ (DB غير جاهز، disk غير مُهيأ) → مُغلَّف في `try/catch` → `var()` fallbacks تعمل

## ٧. Save Flow

```
POST /admin/appearance/brand/settings
        ↓
AppearanceController::updateBrandColors()
        ↓
Validator::make() — hex regex validation لكل حقل
        ↓
AdminBrandThemeSettings::fromArray() — sanitize + validate
        ↓
GeneralSetting::update(['admin_brand_settings' => $brandSettings->toArray()])
        ↓
Cache::forget('general_settings')
        ↓
(new AdminBrandCssGenerator())->generate($brandSettings)
        ↓  يكتب الملف فوراً
Storage::disk('public')->put('admin-theme/admin-brand.css', $css)
        ↓
redirect()->back()->with('ok', '...')
```

التأثير فوري: الطلب التالي للموقع يُحمِّل الـ CSS الجديد.

## ٩. Validation Results

### Primary Color → #0055ff

1. احفظ `primary = #0055ff` من Admin UI
2. `AdminBrandCssGenerator::generate()` يكتب:
   ```css
   --admin-color-primary: #0055ff;
   ```
3. `app.css` يحتوي:
   ```css
   --color-purple-brand: var(--admin-color-primary, #240a37);
   ```
4. النتيجة: `--color-purple-brand` = `#0055ff`
5. `.text-purple-brand`, `.bg-purple-brand` تتغير لـ `#0055ff` ✅

### Secondary Color → #16a34a

1. احفظ `secondary = #16a34a`
2. `--admin-color-secondary: #16a34a;` في CSS
3. `--color-red-brand = #16a34a`
4. `.text-red-brand`, `.bg-red-brand` تتغير ✅

### Backward Compatibility — حذف admin-brand.css

1. `AdminBrandCssGenerator::delete()` (أو `rm storage/app/public/admin-theme/admin-brand.css`)
2. `AdminBrandCssGenerator::publicUrl()` يُعيد `null`
3. `head.blade.php` يتخطى `<link>` بالكامل
4. `app.css`:
   ```css
   --color-purple-brand: var(--admin-color-primary, #240a37);
   ```
   `--admin-color-primary` غير معرَّف → fallback `#240a37` يُستخدم
5. الموقع يعمل بالألوان الافتراضية — لا أخطاء، لا خراب ✅

## ١٢. دعم DesignTokenRegistry في Phase 2

### الحقول المحجوزة حالياً

```css
--admin-color-custom-1: #xxxxxx;   /* custom_1 */
--admin-color-custom-2: #xxxxxx;   /* custom_2 */
…
--admin-color-custom-5: #xxxxxx;   /* custom_5 */
```

### الخطوات المطلوبة في Phase 2

1. **DesignTokenRegistry** — إضافة options جديدة لـ `background_token` و `text_token`:
   ```php
   'custom_1' => 'bg-admin-custom-1',
   'custom_2' => 'bg-admin-custom-2',
   …
   ```

2. **app.css** — إضافة utility classes:
   ```css
   .bg-admin-custom-1  { background-color: var(--admin-color-custom-1); }
   .text-admin-custom-1 { color: var(--admin-color-custom-1); }
   …
   ```

3. **لا تغيير على GeneralSetting** — العمود `admin_brand_settings` يدعم `custom_1..5` بالفعل
4. **لا تغيير على AdminBrandCssGenerator** — يكتب custom colors بالفعل (إذا لم تكن فارغة)

هذا النظام جاهز تماماً لـ Phase 2 — DesignTokenRegistry فقط يحتاج تسجيل الـ CSS class names.
