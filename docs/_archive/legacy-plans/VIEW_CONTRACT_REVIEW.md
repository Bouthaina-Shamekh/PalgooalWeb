## 2. Where `$data` Is Created

### Path A ‚Äî Definition-Driven (via `SectionDefinitionFrontendViewDataFactory::build()`)

```php
// SectionDefinitionFrontendViewDataFactory.php ‚Äî line ~70
$data = $this->normalizeContent($content, $definition, $locale, $translation, $templateKey);
```

`normalizeContent()` merges three sources in priority order:
1. Stored field values from `section_translations.content` (JSON)
2. Default values from `SectionDefinitionField.default_value` for any missing keys
3. The section's `title` from `section_translations.title` as a fallback `title` key

The result is a **flat array** ‚Äî field keys are not split by scope (shared vs. translatable). A `title` field and an `image` field sit at the same level regardless of their scope.

### Path B ‚Äî Legacy/Registered (via `SectionRenderer::renderRegisteredSection()`)

```php
// SectionRenderer.php ‚Äî line ~145
$data = match ($section->type) {
    'features' => self::normalizeFeaturesData($content, $translation),
    default    => $content,
};
```

Same shape ‚Äî a flat array.

## 3. Where `$data` Is Passed to Blade

### `SectionDefinitionFrontendViewDataFactory::build()` ‚Äî **two keys, same value**

```php
// Lines 97-98 (missing template path) and 126-127 (resolved view path)
'data'    => $data,
'content' => $data,   // duplicate alias ‚Äî backward compatibility
```

Both `$data` and `$content` resolve to the **same array** at runtime. The `$content` alias exists for backward compatibility with legacy tenant section templates that read `$content`.

### `SectionRenderer::renderRegisteredSection()` ‚Äî **one key only**

```php
// Line 143
'data'    => $data,
// $content is NOT passed here
```

Legacy registered sections (non-definition-driven) receive only `$data`.

### `tenant/partials/render-sections.blade.php` ‚Äî **completely separate path**

```php
// Passed via @includeFirst for legacy (non-definition-driven) tenant sections
'content' => $content,   // raw JSON array from section_translations
// $data is NOT passed here
```

Tenant sections that fall through the definition-driven path receive `$content` directly from the raw DB query, bypassing `SectionDefinitionFrontendViewDataFactory` entirely.

## 5. The Monaco Scaffold ‚Äî A Critical Hidden Inconsistency

This is the most important finding of this review.

The Monaco editor's "Scaffold ŸÖŸÜ ÿßŸÑÿ≠ŸÇŸàŸÑ" button generates code using variables that **do not exist at runtime**:

```javascript
// generateSnippet() in edit.blade.php
var src = scope === 'translatable'
    ? "$translatableData['" + k + "'] ?? ''"   // ‚Üê NOT a runtime variable
    : "$sharedData['" + k + "'] ?? null";       // ‚Üê NOT a runtime variable
```

The full scaffold `@php` block generated by "Scaffold (ÿßÿ≥ÿ™ÿ®ÿØÿßŸÑ)" looks like:
```blade
@php
    // Scaffold: hero_campaign ‚Äî 2026-06-15
    $title = trim((string)($translatableData['title'] ?? ''));   // ‚Üê undefined
    $image = \App\Support\Sections\SectionFrontendMediaResolver::resolve($sharedData['image'] ?? null); // ‚Üê undefined
@endphp
```

At runtime, `$sharedData` and `$translatableData` are **never defined** by `SectionDefinitionFrontendViewDataFactory`. Only `$data` (and `$content` as its alias) exist.

**Any Blade file written using the scaffold "Scaffold (ÿßÿ≥ÿ™ÿ®ÿØÿßŸÑ)" or "Scaffold ŸÉÿßŸÖŸÑ" button will fail at runtime** with an `Undefined variable $sharedData` error, or silently output nothing (PHP 8 does not throw for null coalescing on undefined variables in non-strict mode, but `trim()` will receive null).

The existing 17 production section files in `front/sections/` were **NOT written using the scaffold** ‚Äî they were hand-written and all use `$data` correctly. The scaffold has never been validated against the actual runtime contract.

This means the `$sharedData`/`$translatableData` naming in the scaffold represents a **planned contract that was never implemented** ‚Äî the scaffold was written with a future split-by-scope design in mind, but the runtime was shipped with a unified flat `$data`.

## 6. Option Comparison

### Option A ‚Äî Keep `$data`

**How it works today:**
```php
// Passed by SectionDefinitionFrontendViewDataFactory
'data'    => $data,    // flat merged array of all fields
'content' => $data,    // alias for backward compatibility
```

In a Blade template:
```blade
@php
    $title = trim((string) ($data['title'] ?? ''));
    $image = \App\Support\Sections\SectionFrontendMediaResolver::resolve($data['image'] ?? null);
    $items = is_array($data['items'] ?? null) ? $data['items'] : [];
@endphp
```

| | |
|---|---|
| **Files to change** | 0 |
| **Runtime impact** | None |
| **Scaffold impact** | None (scaffold is already broken regardless) |
| **Documentation** | Write the contract in `07-section-definitions.md` and `21-developer-guide.md` |
| **Risk** | None |

---

### Option B ‚Äî Migrate to `$fields`

**Would look like:**
```php
// After migration
'fields'  => $data,   // renamed
'content' => $data,   // keep alias
// OR remove 'data' entirely
```

In a Blade template:
```blade
@php
    $title = trim((string) ($fields['title'] ?? ''));
    $image = \App\Support\Sections\SectionFrontendMediaResolver::resolve($fields['image'] ?? null);
@endphp
```

| | |
|---|---|
| **Files to change** | 18+ (17 `front/sections/` + `SectionDefinitionFrontendViewDataFactory.php` + `SectionRenderer.php`) |
| **Runtime impact** | Requires Phase 1 dual-key bridge. Any missed file causes silent null output or PHP error |
| **Scaffold impact** | Also needs update ‚Äî but scaffold is already generating wrong variables (`$sharedData`) |
| **Documentation** | `07-section-definitions.md`, `09-rendering-flow.md`, `21-developer-guide.md`, `CLAUDE.md` |
| **Risk** | Medium. The bridge phase (dual-key `$data` + `$fields`) must be maintained until all files are updated |

**Does `$fields` solve the real problem?** No. The clarity problem is not `$data` vs `$fields`. The real clarity problem is that `$data` is a flat array of mixed shared+translatable fields, and developers cannot tell from the variable name which fields are locale-specific. `$fields` has the same problem ‚Äî it is still a flat merged array. The Monaco scaffold already identified this problem by splitting into `$sharedData`/`$translatableData`, but never wired it up to the runtime.

---

## 7. Recommendation

```
Keep $data  (Option A)
```

**Reasons:**

**1. `$fields` does not solve the actual developer confusion.** The confusion is about shared vs. translatable scope, not about what the variable is called. A developer looking at `$fields['title']` has the same question as `$data['title']`: "Is this locale-specific? Where does it come from?" The name `$fields` is not more informative than `$data` in this context.

**2. Renaming carries real risk with no functional gain.** 17 production files and 96 call sites work correctly today. A rename requires a migration plan, dual-key bridge period, and coordinated update across all files. The benefit is cosmetic.

**3. The Monaco scaffold inconsistency is the real problem.** The scaffold generates `$sharedData`/`$translatableData` which do not exist at runtime. This is a live developer tooling bug that affects any developer using the "Scaffold" button. Fixing this is higher priority than renaming `$data` to `$fields`.

**4. The real architectural decision is `$data` vs `$sharedData + $translatableData`.** If we want to make the contract more expressive, the meaningful split is by field scope ‚Äî not by a generic rename. This is a separate, larger decision that should go through a proper ADR.

## 9. Required Follow-Up Actions (Regardless of Decision)

These are issues that exist today, independent of the naming question:

**HIGH ‚Äî Monaco Scaffold Bug:**
The "Scaffold" button in the Monaco editor generates code using `$sharedData` and `$translatableData` which are not defined at runtime. Any section Blade file written using the scaffold button will not work. Fix options:
- **Quick fix:** Update `generateSnippet()` and `generateFullScaffold()` in `edit.blade.php` to reference `$data['key']` instead of `$sharedData['key']` and `$translatableData['key']`
- **Full fix:** Add `$sharedData` and `$translatableData` to the runtime viewData by splitting the merged `$data` array by field scope in `SectionDefinitionFrontendViewDataFactory`

**MEDIUM ‚Äî Undocumented `$data` Contract:**
The current Blade view contract (`$data` is a flat array of all field values, `$content` is an alias) is not documented in `07-section-definitions.md`. Add a "View Contract" section to that document.

**LOW ‚Äî `$content` dual-purpose alias:**
`$content` means two different things:
- In definition-driven `front/sections/`: alias for `$data` (same array)
- In tenant legacy sections: the raw `section_translations.content` JSON

This naming overlap can confuse developers reading cross-context code. Not urgent, but worth documenting clearly.

## 10. If The Project Later Adopts Split-Scope Variables

The correct future contract (should an ADR approve it) would be:

```php
// SectionDefinitionFrontendViewDataFactory ‚Äî viewData
'fields'           => $data,           // flat merged (backward compat)
'sharedFields'     => $sharedData,     // scope=shared fields only
'translatableFields' => $translatableData, // scope=translatable, current locale
```

This would:
- Match the Monaco scaffold's mental model
- Give developers explicit locale-scope awareness
- Allow `$fields` as the unified fallback for templates that don't need scope distinction
- Require a proper ADR and phased migration

This is **not the current recommendation** ‚Äî it is presented here as the potential future state if the scaffold design intent is validated and approved.
